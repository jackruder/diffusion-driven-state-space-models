* Overview

This notebook documents component tests and synthetic-data comparisons for
DDSSM using the Hydra-based three-stage CLI. Each experiment is a named preset
in ~ddssm.conf~; train, evaluate, and visualize are independent stages that can
be run in any order provided a checkpoint exists.

** Three-stage pattern

#+BEGIN_SRC sh
# 1. Train
python -m ddssm.app experiment=PRESET

# 2. Evaluate (requires a checkpoint from stage 1)
python -m ddssm.evaluate experiment=PRESET \
    +checkpoint=outputs/<run_dir>/checkpoints/ckpt_latest.pth \
    +csv_path=outputs/<run_dir>/metrics.csv

# 3. Visualize (produces PNGs under the same run_dir)
python -m ddssm.visualize experiment=PRESET \
    +checkpoint=outputs/<run_dir>/checkpoints/ckpt_latest.pth \
    +csv_path=outputs/<run_dir>/metrics.csv
#+END_SRC

Hydra writes all outputs (CSV log, checkpoints, PNGs, ~metrics.json~) under
~outputs/<date>/<time>/~ by default.  Pass ~hydra.run.dir=runs/my_run~ to
override.

** W&B logging

#+BEGIN_SRC sh
# Enable W&B (opt-in)
python -m ddssm.app experiment=PRESET wandb=enabled wandb.project=ddssm
#+END_SRC

W&B is disabled by default (~conf/wandb/disabled.yaml~). Use ~wandb=enabled~ to
activate; group / run names are auto-derived from Hydra so multirun trials
cluster correctly in the W&B UI.

*** JSD analysis (one-step distribution match)

The bimodal DGP has a closed-form one-step conditional density (50/50 mixture
of two Gaussians at ~±step_size~ around ~a·xₜ₋₁~). We compare each model's
one-step forecast distribution against this analytic truth using the
Jensen–Shannon divergence (JSD), and contrast against an LOCF baseline that
predicts ~xₜ = xₜ₋₁~ (the worst-case unimodal-collapse scenario).

This recovers the analysis previously done in the legacy ~bimodal_jsd.py~ /
~plot_bimodal_jsd.py~ scripts.

**** 1. Compute per-model JSD (writes ~bimodal_jsd.npz~ as a side-file)

The ~bimodal_jsd~ metric is registered in the eval registry; override the
existing ~bimodal_*~ eval spec at the CLI to add it. Use ~T_split=T-1~ so the
forecast horizon is strictly one step (the analytic truth is a one-step density).

#+BEGIN_SRC sh
# Gaussian
python -m ddssm.evaluate experiment=bimodal \
    +checkpoint=runs/bimodal/gauss/checkpoints/ckpt_latest.pth \
    +csv_path=runs/bimodal/gauss/metrics.csv \
    'experiment.eval.metrics=[bimodal_jsd]' \
    'experiment.eval.num_samples=1024' \
    'experiment.eval.T_split=63' \
    'experiment.eval.kwargs={bimodal_jsd: {npz_path: bimodal_jsd.npz}}' \
    hydra.run.dir=runs/bimodal/gauss

# Diffusion
python -m ddssm.evaluate experiment=bimodal \
    transition=diffusion \
    +checkpoint=runs/bimodal/diff/checkpoints/ckpt_latest.pth \
    +csv_path=runs/bimodal/diff/metrics.csv \
    'experiment.eval.metrics=[bimodal_jsd]' \
    'experiment.eval.num_samples=1024' \
    'experiment.eval.T_split=63' \
    'experiment.eval.kwargs={bimodal_jsd: {npz_path: bimodal_jsd.npz}}' \
    hydra.run.dir=runs/bimodal/diff
#+END_SRC

The ~kwargs={bimodal_jsd: {...}}~ override is per-metric — the runner forwards
that nested dict as ~**kwargs~ when it calls ~eval_bimodal_jsd~. Each run
writes its own ~bimodal_jsd.npz~ next to ~metrics.json~.

* Appendix

** Adding a new experiment

The configuration package lives at ~src/ddssm/conf/~ and mirrors the
section structure of this document:

| ~experiments/component.py~ | Component Tests (synthetic\_gauss, synthetic\_diffusion)                        |
| ~experiments/synthetic.py~ | Synthetic Experiments (harmonic, bimodal, robot\_2d base presets) |
| ~experiments/kdd.py~       | Real-Data Experiment: KDD Cup 2018                                              |

To add a new experiment:

1. **For a new synthetic family**: add the data-generating process to
   ~src/ddssm/data/synthetic.py~, define eval/viz defaults in
   ~src/ddssm/conf/_eval_viz.py~, and register a single base preset in
   ~experiments/synthetic.py~ with ~_experiment_conf(...)~ (no
   ~transition_conf~).  All transition and training-budget variants are
   then expressed as CLI overrides in this file.
2. **For a new real-data dataset**: follow the pattern in
   ~experiments/kdd.py~.  Since dataset-specific hyper-parameters
   (batch size, steps) differ between Gaussian and Diffusion variants,
   registering one preset per transition is reasonable.
3. Register with ~store(..., group="experiment", name="my_preset")~ and
   add any new eval/viz families to ~src/ddssm/conf/_eval_viz.py~.
4. Add coverage to ~tests/test_experiment_configs.py~ and an entry in
   the Results table above.

** Adding a new metric

1. Write a function with signature ~(ctx: EvalContext) -> dict[str, Any]~.
2. Decorate with ~@register_metric("my_metric")~ in
   ~src/ddssm/eval/metrics.py~.
3. Reference it in an ~EvalSpec(metrics=["my_metric", ...])~.

** Adding a new plot

1. Write a function with signature ~(ctx: PlotContext, save_path: str, **kwargs) -> None~.
2. Decorate with ~@register_plot("my_plot")~ in ~src/ddssm/viz/plots.py~.
3. Reference it in a ~VizSpec~ ~PlotSpec(name="my_plot", ...)~.
