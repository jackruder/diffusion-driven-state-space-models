* How this file works

This org file is the source of truth for every named experiment.
Each headline below holds a Python src block whose ~:tangle~ argument
writes a runnable ~experiments/<name>.py~ file. Tangle the whole
buffer with ~C-c C-v t~; the resulting ~.py~ files are picked up by:

- ~python -m ddssm.app experiment=NAME~ — Hydra CLI.
- ~from experiments.NAME import exp~ — programmatic use in
  notebooks / org src blocks / sweeps.
- ~pytest tests/test_experiment_configs.py~ — instantiate-and-train
  smoke tests for every registered experiment.

Don't edit ~experiments/*.py~ directly; edit this file and re-tangle.

** Layers

#+BEGIN_SRC text
ddssm/builders.py        — Encoder, Decoder, Unet, MLPUnet, Schedule, …  (Conf builders)
experiments/_make.py     — make_experiment, override, run, to_yaml       (composer)
experiments/_bases.py    — harmonic_base, bimodal_base, …                (per-family bases)
experiments/_registry.py — experiment_store = store(group="experiment")  (zen-store)
experiments.org          — THIS FILE: experiment definitions             (source of truth)
experiments/*.py         — tangled output of this file                   (generated)
experiments/log.org      — running experiment log                        (sweeps & results)
#+END_SRC

** The store() idiom

We follow the hydra-zen pattern of an explicit, pre-grouped store:

#+BEGIN_SRC python
# experiments/_registry.py
from hydra_zen import store
experiment_store = store(group="experiment")
#+END_SRC

Every experiment file ends with a visible registration call:

#+BEGIN_SRC python
exp = harmonic_base(GaussTransition())
experiment_store(exp, name="harmonic_gauss")
#+END_SRC

At CLI startup ~ddssm._experiment_registry~ imports every
~experiments/<name>.py~ (triggering its ~experiment_store(...)~
call), then issues a single ~experiment_store.add_to_hydra_store()~
to publish the group into Hydra's ConfigStore. No per-module
~cs.store(...)~ boilerplate; the registration is in the source where
you can see it.

** Deriving variants on the fly

For ad-hoc tweaks that don't deserve a name, use ~override()~. It
returns a fresh experiment with the listed fields changed; CLI
strings parse YAML scalars, dicts let you pass Python objects:

#+BEGIN_SRC python
from experiments.harmonic_diffusion import exp as A
from experiments._make import override, run
from ddssm.builders import MLPUnet

# small tweak
B = override(A, "training.steps=200",
                "model.transition.schedule.sigma_min=0.001")

# loop
for mode in ["harmonic", "bimodal", "robot-basis-pursuit"]:
    exp = override(A, f"data.mode={mode}")
    run(exp, run_dir=f"runs/{mode}")

# python value swap (dict form)
C = override(A, {"model.transition.unet": MLPUnet(channels=64, n_layers=3)})
#+END_SRC

You don't need to come back here just to tweak a value — derive from
an existing tangled experiment via ~override()~ in the log or in a
notebook. Come back here when you want to give a configuration a
*name* (so it can be invoked by the Hydra CLI and tested in pytest).

* Synthetic 1D smoke tests

** synthetic_gauss

LGSSM mode, Gaussian transition, 500 steps. CI smoke test.

#+BEGIN_SRC python :tangle experiments/synthetic_gauss.py
"""LGSSM smoke test + Gaussian transition (500 steps)."""

from ddssm.builders import GaussTransition
from experiments._bases import synthetic_base
from experiments._make import run
from experiments._registry import experiment_store

exp = synthetic_base(GaussTransition())
experiment_store(exp, name="synthetic_gauss")

if __name__ == "__main__":
    run(exp, run_dir="runs/synthetic_gauss")
#+END_SRC
